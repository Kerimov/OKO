# План Phase 3: бизнес-логика Access → портал

Цель: закрыть разрыв между **OKO26-1.mde** и веб-порталом по ключевым процессам методолога.

## Принцип приоритетов

```
3.1 Сальдо  →  3.2 Комплект ZID/EID  →  3.3 Расшифровки  →  3.4 FTotal / Recalc
         ↓
3.5 Личные кабинеты организаций  →  3.6 Агрегация ✅
```

**Не делаем:** обмен файловыми пакетами филиал ↔ головной офис (legacy Access «Сохранить на диск») — все работают в одном веб-портале.

## Этап 3.5 — Личные кабинеты организаций (вместо обмена пакетами) ✅ (базовый контур)

**Цель:** у каждой организации свой изолированный контур в портале: комплект форм, заполнение, проверки — без выгрузки/импорта MDB/ZIP между филиалами.

**Сделано (вариант C — учётные записи):**
- Таблицы `users` / `sessions`, вход по логину/паролю (`POST /api/auth/login`)
- Bootstrap admin: `OKO_BOOTSTRAP_ADMIN_USER` / `OKO_BOOTSTRAP_ADMIN_PASSWORD`
- Изоляция по `zid` для роли org: instances, packages, organizations, periods, work-context
- Портал: `/login`, `/admin/users`, выход, скрытие админ-разделов для org-пользователей
- Совместимость с legacy Bearer-токенами (`OKO_ADMIN_TOKEN` / `OKO_USER_TOKEN`)

**Осталось (следующие итерации):**

| Задача | Статус |
|--------|--------|
| Статусы форм: черновик / сдано | ✅ |
| Админ: просмотр комплектов всех организаций (дашборд) | ✅ `/admin/packages` |
| LDAP / SSO | отдельный этап |

**Оставляем как есть (не legacy-обмен):** экспорт JSON/Excel на `/tools` — резервная копия и отчётность, не канал между филиалами.
