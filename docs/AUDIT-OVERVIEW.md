# Обзор проекта OKO — для аудита

Документ описывает **что было сделано**, **зачем** и **в какой последовательности**. Предназначен для технического аудита, приёмки и передачи проекта новой команде.

**Дата актуализации:** июнь 2026  
**Репозиторий:** `Kerimov/OKO`  
**Статус:** активная разработка, production на Vercel + Render

---

## 1. Назначение проекта

### Исходная система

**ПК «ОКО»** — программный комплекс корпоративной специализированной отчётности (MS Access):

- приложение `OKO26-1.mde` (формы, VBA);
- база `z261.mdb` (76 шаблонов форм, правила увязок, сальдо, справочники);
- ежеквартальная отчётность организаций группы в тысячах рублей.

### Цель веб-проекта

Создать **веб-портал**, который:

1. Заменяет десктопный Access для заполнения и сдачи форм.
2. Хранит данные на сервере (не в файлах на ПК пользователя).
3. Разграничивает доступ: организации видят только свои данные, ЦО — всю группу.
4. Сохраняет бизнес-логику оригинала: увязки, пересчёт, сальдо, агрегация.

### Что не входит в scope

- Обмен файловыми пакетами «филиал ↔ головной офис» (legacy «Сохранить на диск») — все работают в одном портале.
- Полная замена VBA-макросов Access один в один — реализованы ключевые процессы методолога.
- LDAP / SSO — запланировано на следующий этап.

## 2. Этапы разработки

Разработка велась **поэтапно** с фиксацией планов в `docs/PHASE2-PLAN.md` и `docs/PHASE3-PLAN.md`.

### Phase 1 — MVP портала (начало)

| Результат | Описание |
|-----------|----------|
| React-приложение | Каталог 76 форм, редактор таблиц, localStorage |
| JSON-схемы | Генерация из `z261.mdb` скриптом `scripts/generate_schemas_from_mdb.py` |
| PDF-экспорт | Заполненные формы в PDF (pdfmake) |
| Правила | `checks.json`, `saldo-rules.json` — выгрузка из MDB |

**Коммит:** `47d3e8c` — инициализация репозитория.

### Phase 2 — API и метаданные в БД

| Результат | Описание |
|-----------|----------|
| Express API | `server/` — REST, SQLite |
| Редакторы | Увязки, формы, сальдо, Excel — `/admin/*` |
| Движок проверок | `portal/src/engine/checkEngine.ts` — 3600+ правил |
| Fallback | Портал работает без API (JSON + localStorage) |

**Коммит:** `63483af` — SQLite API, checks engine, admin tools.

### Phase 3 — Бизнес-логика Access

| Результат | Описание |
|-----------|----------|
| Комплекты ZID/EID | Организации, периоды, «Завести 76 форм» |
| Сальдо | FormCorrespondence + a_tblsaldo |
| Расшифровки | sp_rash, пороги |
| Пересчёт | FTotal / recalc rules |
| Личные кабинеты | Логин/пароль, роли admin/org |
| Агрегация | Свод дочерних org, ручная агрегация |

**Коммиты:** `96108ce`, `ebd95c7` — packages, recalc, rash, users, aggregation.

### Phase 2.5 — Production

| Результат | Описание |
|-----------|----------|

| Docker | `Dockerfile`, `docker-compose.yml` |
| Деплой | Vercel (портал) + Render (API) — `docs/DEPLOY.md` |

### Доработки UX и документации (2026)

## 4. Роли и безопасность

### Роли

| Роль | Доступ |
|------|--------|
| **admin** | Все организации, редакторы метаданных, пакетные операции, пользователи, аудит |
| **org** | Только своя организация (ZID): формы, комплект, каталог, настройки |

### Авторизация

1. **Учётные записи** — логин/пароль, сессия 7 дней (`OKO_BOOTSTRAP_ADMIN_*` при первом запуске).
2. **Bearer-токены** (legacy) — `OKO_ADMIN_TOKEN`, `OKO_USER_TOKEN` для сервисных сценариев.

Разграничение на API: `server/src/orgScope.ts` — фильтрация по ZID пользователя.

### Аудит

Журнал изменений **метаданных** (шаблоны, правила, пользователи): `server/src/audit.ts`, UI `/admin/audit`.  
Изменения ячеек форм логируются через `updated_at` экземпляра.

## 6. Развёртывание (production)

```
Пользователь → Vercel (portal/) → HTTPS
                    ↓ VITE_API_URL
              Render (server/) → PostgreSQL
```

- Портал: статика, Root Directory `portal`.
- API: Docker-образ из корня репозитория, `DATABASE_URL` от managed PostgreSQL.
- Данные **не теряются** при redeploy API (БД отдельно от контейнера).

Инструкция: [DEPLOY.md](DEPLOY.md).

---

## 7. Как проверить работоспособность

1. `curl https://<api>/api/health` → `{"ok":true,"db":"postgresql",...}`
2. Вход в портал под admin → разделы Комплект, Сводка и импорт, Пользователи.
3. Создать org-пользователя → войти под ним → видны только формы своей организации.
4. Завести комплект → заполнить форму → Проверить → Сдать.
5. Раздел **Инструкция** — встроенное руководство.
