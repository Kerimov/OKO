# ЧАСТЬ I. СПИЧ: ЗАЩИТА СТЕКА (ТЗ v1.2) И СРАВНЕНИЕ С 1С

> Полный текст из `docs/Спич-защита-стека-ОКО-и-сравнение-с-1С.docx`

**Контекст:** импортозамещение Microsoft Access‑системы (MDE+MDB), масштаб 1000+ организаций, портал ЦО/организаций + офлайн‑дочки в сетевой папке, версионируемая методология (формы/увязки/сальдо/rash/Excel), обмен ReportPackage JSON.

---

## 1. Короткий спич на 2–3 минуты

Мы не просто «переписываем формы», мы переносим методологию из Access в промышленную архитектуру: метаданные форм, увязки, правила пересчёта, сальдо и расшифровки становятся **данными в БД**, а портал и десктоп — удобными клиентами к этим правилам.

Поэтому стек в ТЗ выбран под ключевые риски Access: масштаб, безопасность, офлайн‑режим и сопровождение методологии каждый квартал.

- **Портал**: React + TypeScript (сложные табличные формы, быстрый UI для ЦО и организаций).
- **API**: NestJS (основной) или Fastify (альтернатива) на TypeScript — единые контракты, RBAC/аудит/транзакции.
- **СУБД**: PostgreSQL / Postgres Pro в проде — надёжность, масштабирование, бэкапы, импортозамещение.
- **Офлайн**: Tauri 2 (Rust) + React + native SQLite (WAL) — работа без интернета в сетевой папке комплекта.
- **Обмен**: ReportPackage JSON — версионированный контракт между офлайном и ЦО.

Ключевой эффект: один доменный движок (`@oko/engine`) исполняет правила одинаково везде, что снижает риск расхождений с Access и упрощает приёмку на эталонных комплектах.

## 2. Позиция «стек vs 1С»

**Если у Заказчика «зоопарк»** и задача — сделать именно систему из ТЗ (табличные формы + офлайн + движки правил), то:

- **Дешевле и быстрее** — на нашем стеке (React/TS + Nest/Fastify + Postgres + офлайн‑клиент).
- **1С будет дешевле** только при существенном упрощении требований (например, без офлайна/без редакторов методологии) и при сильной внутренней 1С‑команде.

## 3. Готовые ответы на возражения (Q&A)

### Возражение: «Почему JSON, а не XML?»

ReportPackage в ТЗ — это не «документ с разметкой», а **пакет структурированных данных** (мета + список экземпляров + опционально правила). Для нашего стека (TypeScript/React/Nest/Fastify) JSON — нативный формат: быстрее и дешевле реализовать сериализацию/парсинг, проще версионировать контракт, удобнее отлаживать и делать автотесты (diff пакетов с эталоном).

- **Дешевле разработка**: меньше инфраструктурного кода под преобразования и схемы.
- **Проще эволюция**: добавление полей и версий контракта без поломки старых клиентов.
- **Проще поддержка**: легче читать, сравнивать, диагностировать ошибки импорта/экспорта.

XML можно поддержать **как внешний формат интеграции** (если потребуется корпоративными регламентами), но внутренний обмен пакетами в системе рациональнее оставить на JSON.

### Возражение: «Почему не сделать всё на 1С? Это же привычно»

1С сильна как платформа учёта и типовых регламентов, но ТЗ про другое: сложные табличные формы «как в Excel», версионируемую методологию (3600+ правил), офлайн‑контур в сетевой папке и обмен JSON‑пакетами. На нашем стеке эти сценарии реализуются быстрее и с меньшими компромиссами по UX и сопровождению.

### Возражение: «У нас зоопарк. Не станет ли ещё одним зоопарком?»

Наоборот: мы выносим отчётность в отдельный контур с чёткими границами — портал + API + БД + офлайн‑клиент, с единым форматом обмена ReportPackage. Это снижает связанность с остальными системами и упрощает сопровождение.

### Возражение: «TypeScript/веб — это несерьёзно для “бухгалтерии”»

Серьёзность обеспечивается не «языком», а архитектурой: промышленная СУБД (Postgres Pro), транзакции, аудит, RBAC, SSO, резервирование, регламентный деплой. TypeScript используется как прикладной слой и доменная логика, которая хорошо тестируется и быстро развивается.

### Возражение: «Как гарантируете совпадение с Access?»

ТЗ фиксирует подход: не reverse VBA из .mde, а воспроизводим методологию из MDB/регламентов и сверяемся на эталонных комплектах. Движки правил (увязки, сальдо, пересчёт, rash) — единые, с автотестами и версионированием.

### Возражение: «Офлайн в сетевой папке — риск (SMB/коллизии)»

Да, риск признан в ТЗ, поэтому выбрана модель: native SQLite + WAL, ограничение числа клиентов, блокировки ячеек/presence, и обязательные бэкапы. Это более управляемо, чем общий MDB в Access.

### Возражение: «Безопасность: SSO, роли, аудит»

В стеке это штатно закрывается: Keycloak/LDAP/SSO, RBAC на API (guards), аудит изменений ячеек и правил, TLS через Nginx/Angie, подпись релизов и SBOM (как в требованиях импортозамещения).

### Возражение: «Импортозамещение — где гарантии?»

В ТЗ заложено: сервер на Astra/ALT/РЕД ОС, СУБД Postgres Pro / PostgreSQL, reverse proxy Nginx/Angie, без облачных сервисов в prod, внутренние зеркала зависимостей, подпись ПО. Это делает решение переносимым on‑prem и соответствующим политике импортозамещения.

### Возражение: «Сколько будет стоить сопровождение?»

Снижаем стоимость владения тем, что методология становится данными в БД и обновляется через редакторы, а не через «пересборку монолита». Плюс стандартный DevOps‑контур: Docker‑деплой, миграции БД, мониторинг, бэкапы, версионирование API/контрактов.

### Возражение: «Почему NestJS, а не Fastify / наоборот?»

ТЗ допускает оба варианта. Для большого количества модулей и долгого сопровождения удобнее NestJS (строгая структура и паттерны безопасности). Fastify — допустимая альтернатива для более лёгкого API и быстрого MVP. Ключевое: контракты и доменная модель фиксируются так, чтобы смена каркаса не затрагивала фронт и данные.

### Возражение: «Лицензии 1С могут быть дешевле/дороже — как сравнивать?»

В нашем стеке основной капекс — разработка и инфраструктура (сервер/БД/подписи), а не лицензии прикладной платформы. В 1С часто добавляются лицензии платформы, серверов, клиентских доступов, плюс стоимость редких специалистов под сложный UI. При «зоопарке» и необходимости офлайна/табличного UX обычно дешевле сделать отдельный контур на web+desktop.

# ЧАСТЬ II. НАЗНАЧЕНИЕ ПРОЕКТА И КОНТЕКСТ

## Исходная система (Access)

**ПК «ОКО»** — программный комплекс корпоративной специализированной отчётности на MS Access:

| Компонент | Содержание |
|-----------|------------|
| `OKO26-1.mde` | UI + VBA (скомпилированное приложение) |
| `z261.mdb` | Данные, метаданные, правила |
| 76+ форм | N01–N19, ND — ежеквартальная отчётность |
| `a_tblchecks` | ~3610 правил увязок |
| `a_tblsaldo` | ~2853 правил переноса сальдо |
| `a_stblROWs` / `a_stblFIELDs` | Конструктор строк и граф (~2879 / ~1000) |

Пользовательские сценарии из руководства: выбор организации/подразделения; «Завести пустые формы»; перенос сальдо; заполнение с расшифровками по порогам; проверка увязок (F1); «Сохранить на диск»; «Принять» / «Принять частично»; агрегация; сетевой режим с общей MDB.

## Цель веб-проекта OKO

Создать систему, которая:

1. Заменяет десктопный Access для заполнения и сдачи форм.
2. Хранит данные на сервере (не в файлах на ПК пользователя).
3. Разграничивает доступ: организации видят только свои данные, ЦО — всю группу.
4. Сохраняет бизнес-логику оригинала: увязки, пересчёт, сальдо, агрегация, расшифровки.
5. Поддерживает офлайн-контур дочерних организаций (десктоп «ОКО Заполнение»).

## Среды и порты

| Среда | Портал | API |
|-------|--------|-----|
| Production | Vercel — статика из `portal/` | Render — `server/`, PostgreSQL |
| Локально | http://localhost:5173 | http://localhost:3001 |

## Быстрый старт разработки

Скрипт `dev.sh` в корне репозитория запускает API и портал параллельно:

```bash
./dev.sh
# API:    http://localhost:3001  (NestJS)
# Portal: http://localhost:5173
# Ctrl+C — останавливает оба процесса
```

Вручную:

```bash
cd server-nest && npm install && npm run dev   # :3001
cd portal && npm install && npm run dev        # :5173
```

## Общая схема потоков данных

```
┌─────────────────────────────────────────────────────────────┐
│  Браузер / Electron                                         │
│  portal/ — React SPA (Vite)                                 │
│  ┌─────────────┐  ┌──────────────┐  ┌─────────────────────┐ │
│  │ pages/      │  │ engine/      │  │ storage.ts          │ │
│  │ UI экраны   │  │ бизнес-логика│  │ API ↔ localStorage  │ │
│  └─────────────┘  └──────────────┘  └─────────────────────┘ │
└───────────────────────────┬─────────────────────────────────┘
                            │ HTTPS  /api/*
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  server/ — Express API (TypeScript)                         │
│  auth · instances · forms · checks · packages · users     │
└───────────────────────────┬─────────────────────────────────┘
                            │
              ┌─────────────┴─────────────┐
              ▼                           ▼
     PostgreSQL (prod)              SQLite (dev)
     data/schema.postgresql.sql     data/schema.sql
```

**Ключевые принципы:**

1. Портал может работать автономно (localStorage + JSON в `public/`), но в production всегда подключён к API.
2. Бизнес-логика (увязки, пересчёт и т.д.) живёт в `portal/src/engine/` и выполняется **на клиенте** (браузер / Electron). Сервер хранит данные и отдаёт их через API.
3. Десктоп переиспользует те же движки через alias `@portal` — один код проверок для веба и офлайна.

## Поток данных: заполнение и проверка формы

```
1. Пользователь открывает /my/:id
2. storage.ts → GET /api/instances/:id
3. FormTable рендерит ячейки из schema + values
4. Редактирование → PUT /api/instances/:id
5. «Проверить форму» → checkEngine (клиент) + snapshot всех форм периода
6. «Сдать» → PATCH status=submitted
```

## Авторизация

```
Запрос → authMiddleware
           ├─ Bearer token → role: admin | user
           └─ Session cookie → user: { role, zid, ... }
                └─ orgScope → фильтр WHERE zid = user.zid
```

| Роль | Доступ |
|------|--------|
| **admin** | Все организации, редакторы метаданных, пакетные операции, пользователи, аудит |
| **org** | Только своя организация (ZID): формы, комплект, каталог, настройки |

# ЧАСТЬ V. УВЯЗКИ — ПОЛНОЕ ОПИСАНИЕ

## 5.1. Что такое увязка

**Увязка** — правило контроля согласованности данных между формами одного комплекта отчётности. Наследие таблицы Access `a_tblchecks`.

В системе:

| Показатель | Значение |
|------------|----------|
| Всего правил в `checks.json` | 3600 |
| С `periodActive: true` | ~160 (участвуют в обычной проверке формы) |
| С `active: true` | ~160 |
| С `forAggrOnly: true` | отдельный набор для агрегации |
| Источник | экспорт из `z261.mdb` → `portal/public/data/checks.json` |

Формат выражения — VBA-подобный, как в оригинальном Access:

```
Cell("N01_01","G",0200) = Cell("N02_2","P",40)
```

По ТЗ (раздел 4.1.4): хранение ~3600 правил в БД, редактор, парсер Cell/CellK/TOTAL, проверка на клиенте (отзывчивость).

---

## 5.2. Где хранятся правила

### В БД (model)

Таблица `check_rules` — поля:

| Колонка SQL | Поле JSON/DTO | Описание |
|-------------|---------------|----------|
| `number` | `number` | Уникальный номер правила |
| `expression` | `expression` | Основное выражение |
| `expression_alt` | `expressionAlt` | Продолжение (Access разбивал длинные правила на 2 поля) |
| `message` | `message` | Текст ошибки для пользователя |
| `for_aggr_only` | `forAggrOnly` | Только для проверки после агрегации |
| `first_level` | `firstLevel` | Первый уровень иерархии |
| `active` | `active` | Правило активно |
| `period_active` | `periodActive` | Участвует в проверке периода (основной режим) |
| `period` | `period` | Периодичность: `год`, квартал и т.д. |
| `info` | `info` | Служебная информация |

### Первичная загрузка

`server/src/checks.ts`:

1. `seedCheckRulesFromJson()` — при первом bootstrap, если таблица пуста, импортирует из `portal/public/data/checks.json`.
2. `reimportCheckRulesFromJson()` — полная перезаливка (DELETE + INSERT) через API `POST /api/checks/reimport`.

### Загрузка в движок (клиент)

`portal/src/api.ts` → `loadChecks()`:

1. Пробует `GET /api/checks/export` (из БД).
2. Fallback: `portal/public/data/checks.json` (если API недоступен).

---

## 5.3. Где выполняется проверка

**Важно:** проверка увязок выполняется **на клиенте** (браузер или Electron), а не на сервере.

| Роль | Что делает |
|------|------------|
| **Сервер** | Хранит правила в `check_rules`, отдаёт через `/api/checks/export`; хранит данные форм |
| **Клиент** | Загружает правила + все экземпляры периода, вычисляет выражения, показывает результат |

Клиентский прогон в UI остаётся для отзывчивости. Перед сдачей (`PATCH …/status` → `submitted`) сервер выполняет period-проверки через `@oko/engine` (`server/src/instance-submit.ts`) и отклоняет сдачу при ошибках (422).

## 5.4. Функции в выражениях

| Функция | Синтаксис | Смысл | Реализация |
|---------|-----------|-------|------------|
| `Cell` | `Cell("FormId","Column",RowNo)` | Значение одной ячейки | `evalContext.getCell()` → `buildFormIndex` |
| `CellK` | `CellK("FormId","Col","=условие","ключСтроки")` | Сумма по строкам с фильтром | `getCellKValue()` в `checkRunCore.ts` |
| `TOTAL` | `TOTAL("FormId","Column")` | Сумма по всей графе формы | `evalContext.getTotal()` |
| `Clng` | `Clng(выражение)` | Округление до целого | `Math.round()` в `cellExpression.ts` |

Поддерживаются:

- арифметика: `+`, `-`, `*` (с приоритетом умножения);
- логика: `AND`, `OR` (слева направо, short-circuit);
- скобки;
- сравнения: `=`, `<>`, `<`, `>`, `<=`, `>=`.

**Допуск при сравнении:** `0.005` (как в Access для округлений в тысячах рублей).

```typescript
// cellExpression.ts — compareValues
case "=":  return Math.abs(left - right) <= tolerance;   // 0.005
case "<>": return Math.abs(left - right) > tolerance;
case ">=": return left >= right - tolerance;
case "<=": return left <= right + tolerance;
```

## 5.6. Фильтрация правил (`pickRules`)

Код: `portal/src/engine/checkRunCore.ts` (и дубликат в `checkEngine.ts` для оркестрации).

```typescript
function pickRules(checks, options) {
  return checks.filter((c) => {
    if (!c.expression) return false;
    const mode = options.mode ?? "period";
    if (mode === "period" && !c.periodActive) return false;
    if (mode === "active" && !c.active && !c.periodActive) return false;
    if (options.excludeAggr && c.forAggrOnly) return false;
    if (options.formId) {
      const full = combineCheckExpression(c.expression, c.expressionAlt);
      if (!expressionUsesForm(full, options.formId)) return false;
    }
    return true;
  });
}
```

**Lazy-load форм:** `formsUsedByFormChecks(checks, formId)` собирает Set всех formId, упомянутых в правилах текущей формы (через `extractCellRefs`, `extractCellKRefs`, regex TOTAL). Десктоп использует это, чтобы не грузить все 76 форм, а только нужные.

## 5.7. Склейка `expression` + `expressionAlt`

В Access длинные правила разбивались на два поля БД. Функция `combineCheckExpression()` в `cellExpression.ts`:

```typescript
export function combineCheckExpression(expression, expressionAlt) {
  const base = expression.trim();
  if (!expressionAlt?.trim()) return base;
  const alt = expressionAlt.replace(/\r\n/g, " ").replace(/\s+/g, " ").trim();
  const isContinuation =
    /Cell\s*\(|CellK\s*\(/i.test(alt) ||
    /^[+(\-]/i.test(alt) ||
    /^(and|or)\b/i.test(alt) ||
    /\b(and|or)\s*$/i.test(base) ||
    /[+\-]$/.test(base);
  if (!isContinuation) return base;
  return `${base} ${alt}`.trim();
}
```

Если `expressionAlt` начинается с `+`, `-`, `and`, `or`, `Cell(`, `CellK(` — склеивается с основным. Иначе используется только `expression`.

## 5.8. Парсер выражений (`cellExpression.ts`)

### Нормализация

```typescript
normalizeCheckExpression(expr):
  AND/OR → " and " / " or "
  схлопывание пробелов
  "123and" → "123 and "
```

### Парсинг булевой части

`parseBoolExpr` строит дерево:

- `or` — разбиение по top-level `or`
- `and` — разбиение по top-level `and`
- `cmp` — сравнение: `splitCompare` ищет `<=`, `>=`, `<>`, `<`, `>`, `=` вне скобок и строк

### Парсинг арифметики

`tokenizeArithmetic` → `parseArithmetic`:

- `Cell(...)`, `CellK(...)`, `TOTAL(...)`, числа, `+`, `-`, `*`, скобки
- приоритет: `*` выше `+`/`-`

### Вычисление

`evaluateBoolNode`:

- `and`: все части должны быть true; при первом false — возврат с `failedClause`
- `or`: хотя бы одна true; если все false — последний fail
- `cmp`: `left = evaluateExpr(parseArithmetic(left))`, `right = ...`, `compareValues(op, left, right, 0.005)`

## 5.9. Чтение ячеек (`getCell`)

```typescript
getCell("N01_01", "G", 200)
```

1. `buildFormIndex(instances)` → `Map<"N01_01", Map<"200", RowData>>`
2. Ключ строки: `String(row.num).trim()` — `0200` в выражении парсится как `parseInt("0200")` → `200`
3. `rowData["G"]` — значение графы G
4. `parseCellValue`: пустое/null/"" → **0**; запятая → точка; пробелы убираются

Если форма не загружена или строка отсутствует → **0**.

## 5.10. CellK — сумма с фильтром

```typescript
getCellK("N14_1", "C", "=25100", "24_839")
```

`getCellKValue(rows, column, condition, rowKey)`:

1. Для каждой строки формы N14_1:
2. `rowMatchesCondition(row, "=25100")` — проверяет `row.num`, `row.code`, `row.account` против условия
3. `rowMatchesKey(row, "24_839")` — проверяет `row.code`, `row.account`, `row.name`, `row.num` (точное совпадение или подстрока)
4. Если оба условия → `sum += parseCellValue(row["C"])`
5. Если ни одна строка не подошла → **0** (не ошибка, а нулевая сумма)

## 5.14. Типичные причины падения увязок

| Причина | Что происходит в коде | Как диагностировать |
|---------|----------------------|---------------------|
| Форма не заполнена | `getCell` → 0, сравнение с ненулевым | left/right = 0 в результате |
| Другая форма не сохранена | экземпляр отсутствует → все её Cell = 0 | проверить наличие формы в `/my` |
| Округление | допуск 0.005 обычно спасает | разница < 0.005 → passed |
| CellK не нашёл строку | сумма 0 | left/right, проверить num/code/account строк |
| Правило выключено | `periodActive: false` → не в pickRules | правило не в списке результатов |
| Синтаксис | `parseError: true`, `skipped++` | error в item, CheckParseError |
| forAggrOnly | исключено в обычной проверке | нужен runAggregationChecks |

# ЧАСТЬ VI. ПРИМЕРЫ ПРАВИЛ — ПОШАГОВЫЙ РАЗБОР

## Пример 1 — простая кросс-форменная увязка (правило №50)

**Из `checks.json`:**

```json
{
  "number": 50,
  "expression": "Cell(\"N01_01\",\"G\",0200)=Cell(\"N02_2\",\"P\",40)",
  "expressionAlt": null,
  "message": null,
  "forAggrOnly": false,
  "firstLevel": false,
  "active": false,
  "periodActive": true,
  "period": null,
  "info": null
}
```

### Бизнес-смысл

| Ссылка | Форма | Строка | Графа | Содержание (из schema) |
|--------|-------|--------|-------|------------------------|
| Левая часть | N01_01 | 200 | G (Кредит) | «Амортизация основных средств, инвестиционной недвижимости…» |
| Правая часть | N02_2 | 40 | P | «САЛЬДО на начало отчетного периода» |

Увязка: кредит по амортизации в балансе должен совпасть с показателем в форме основных средств.

### Что происходит при проверке формы N01_01

**Шаг 1. Фильтрация** (`pickRules`):
- `periodActive === true` → правило №50 проходит
- `forAggrOnly === false` → не исключается
- в выражении есть `"N01_01"` → правило относится к этой форме

**Шаг 2. Загрузка данных:**
- нужны экземпляры **N01_01** и **N02_2**
- `formsUsedByFormChecks` вернёт `Set { "N01_01", "N02_2" }`
- если у шаблона несколько версий — `latestInstancePerTemplate` берёт последнюю по `updatedAt`

**Шаг 3. Чтение ячеек:**

```typescript
left  = getCell("N01_01", "G", 200)   // 0200 → parseInt → 200
right = getCell("N02_2", "P", 40)
```

**Шаг 4. Сравнение:**

```
compareValues("=", left, right, 0.005)
// Math.abs(left - right) <= 0.005
```

**Шаг 5. Результат при расхождении:**
- N01_01 G/200 = 1 500 000
- N02_2 P/40 = 1 480 000
- разница 20 000 > 0.005 → **failed**
- `failedClause`: `Cell("N01_01","G",0200) = Cell("N02_2","P",40)`
- подсветка: ключ `200:G` в FormTable

## Пример 2 — CellK, несколько форм (правило №467)

```json
{
  "number": 467,
  "expression": "Cell(\"N13_1\",\"C\",420)=CellK(\"N14_1\",\"C\",\"=11650\",\"24_922\")+CellK(\"N14_1\",\"C\",\"=24000\",\"24_919\")",
  "message": "Расхождение объяснить",
  "periodActive": false
}
```

### Бизнес-смысл

Сумма в N13_1 (строка 420, графа C) должна равняться сумме двух выборок из N14_1 по условиям на счёт/код.

### Пошаговый расчёт

Допустим, в N14_1:

| num | code | C |
|-----|------|---|
| 11650 | 24_922 | 800 |
| 24000 | 24_919 | 200 |
| 11650 | 24_999 | 500 | ← rowKey "24_922" не совпал, не входит |

```
left  = Cell("N13_1","C",420)                    = 950
right = CellK(..., "=11650", "24_922")           = 800
      + CellK(..., "=24000", "24_919")           = 200
      = 1000

950 ≠ 1000 → failed
message: «Расхождение объяснить»
```

### Важно про periodActive

`periodActive: false` — **не попадёт** в обычную проверку (`mode: "period"`). Сработает в режиме `all` или если включить `periodActive` в редакторе `/admin/checks`.

## Пример 3 — AND + CellK (правило №99)

```
CellK("N14_1","C","=25100","24_839")
  <= Cell("N04_5","I",80) - Cell("N04_5","I",20)
AND
Cell("N04_5","I",80) - Cell("N04_5","I",20) >= 0
```

`message`: «Расхождение объяснить»

### Формы

N14_1, N04_5 — обе должны быть в комплекте.

### Вычисление

```
diff  = Cell("N04_5","I",80) - Cell("N04_5","I",20)   // напр. 500
cellK = CellK("N14_1","C","=25100","24_839")          // напр. 600
```

AND проверяется **слева направо**, при первом fail — стоп:

| Часть | Условие | Результат |
|-------|---------|-----------|
| 1 | `600 <= 500` | **false** → правило падает |
| 2 | `500 >= 0` | не вычисляется |

Результат: `failedClause = 'CellK(...) <= Cell(...)-Cell(...)'`, `left=600`, `right=500`.
