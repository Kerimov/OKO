Техническое задание на разработку программного комплекса корпоративной специализированной отчётности «ОКО»
Документ определяет требования к созданию импортозамещённой системы, заменяющей программный комплекс ПК «ОКО» на базе Microsoft Access. Версия: 1.2. Дата: июнь 2026.
1. Общие сведения
1.1. Наименование системы
Полное наименование: программный комплекс корпоративной специализированной бухгалтерской отчётности «ОКО». Условные обозначения: ПК «ОКО», Система, Портал ЦО, Клиент заполнения.
1.2. Заказчик и исполнитель
Заказчик: _________________________________ Исполнитель: _________________________________
1.3. Основание для разработки
Необходимость импортозамещения ПО на базе Microsoft Access (32-bit Office 2000–2013).
Требование обеспечить заполнение, проверку и сдачу комплектов отчётности для большого числа организаций группы.
Регламенты и методология ПК «ОКО» (руководство пользователя, письма методологов).
Эталонный комплект Access: OKO26-1.mde + z261.mdb (папка 12345 / ежеквартальный архив OKO26-k.rar).

2. Назначение и цели создания системы
2.1. Назначение
Система предназначена для подготовки, проверки, агрегации и передачи корпоративной специализированной отчётности (76+ форм разделов N01–N19, ND) взамен ПК «ОКО» на Microsoft Access. Система должна поддерживать:
централизованный контур (ЦО): методология, приём комплектов, агрегация, администрирование;
контур организаций: заполнение форм, проверки, расшифровки, экспорт в ЦО;
офлайн-контур дочерних организаций без доступа к веб-порталу: совместное заполнение в сетевой папке и обмен JSON-комплектом.

4. Требования к системе
4.1. Функциональные требования
4.1.1. Организации, периоды, комплекты (ZID/EID)
Ведение справочника организаций (ZID), периодов (EID), иерархии головная/дочка.
Создание комплекта отчётности на период: 76 пустых экземпляров форм.
Контекст работы: выбранные ZID и EID для пользователя.
Дашборд полноты комплекта: заполнено / черновик / сдано.
4.1.2. Заполнение форм
Рендер формы по метаданным (строки, графы, типы, итоги FTotal).
Редактирование ячеек, реквизиты (организация, период, ед. изм. — тыс. руб.).
Подписи ответственных лиц.
Статусы экземпляра: draft / submitted.
Сохранение на уровне ячеек (form_cell_values) с аудитом.
4.1.3. Конструктор форм (метаданные)
CRUD строк (a_stblROWs) и граф (a_stblFIELDs) через UI администратора.
Импорт/экспорт схем из MDB/JSON.
Превью формы при изменении метаданных.
Версионирование изменений метаданных.
4.1.4. Увязки (проверки)
Хранение ~3600 правил в БД (check_rules).
Редактор: выражение, сообщение, флаги active, period_active, for_aggr_only.
Парсер и исполнитель выражений Cell(), CellK(), TOTAL() и сравнений.
Проверка на клиенте (отзывчивость) и на сервере (обязательная перед сдачей).
Тест выражения на данных выбранного периода.
4.1.5. Пересчёт итоговых строк
Правила FTotal / row-formulas из MDB.
Автопересчёт при изменении ячеек (настраиваемо).
4.1.6. Перенос сальдо
Двухэтапный перенос: из предыдущего квартала и из аналогичного периода прошлого года.
Режим «только проверить» с отчётом расхождений.
Правила из a_tblsaldo и FormCorrespondence (Yellow/Red/Blue/Green).
UI аналог «Проверка или перенос входящих остатков».
4.1.7. Контрагенты и расшифровки
Справочник sp_kontr: INN, KPP, OGRN, GUID ОБДНСИ, тип (внутригрупповой/ассоциированный/внешний).
Фильтры «Показать» по форме (ограниченные списки).
Пороги расшифровки и значения «ПРОЧИЕ», «ФИЗИЧЕСКИЕ ЛИЦА».
Подформы расшифровок (sp_rash, t_ras); проверки rashEngine.
Форма N99 — изменения справочника контрагентов.
4.1.8. Агрегация и приём комплектов
Частичный приём форм из файла другого исполнителя.
Полный приём комплекта.
Агрегация: суммирование ячеек нескольких подразделений (a_tblAgg_List).
Контроль пересечения форм при частичном приёме.
4.1.9. Импорт/экспорт
ReportPackage JSON (версионированный контракт, включая rules при необходимости).
Импорт комплекта в ЦО (аналог «Сводка и импорт»).

4.1.10. Администрирование
Пользователи, роли, права на формы и операции.
Журнал аудита (изменения ячеек, правил, метаданных).
Резервное копирование БД комплекта.
Режим «только чтение» комплекта после сдачи.
Настройки периода, шрифтов, путей экспорта (аналог tblConfig).
4.1.11. Десктоп «ОКО Заполнение» (офлайн)
Работа без интернета; данные в сетевой папке комплекта.

6.1. Альтернативный вариант backend на Fastify
Fastify рассматривается как допустимая альтернатива NestJS для случая, когда система трактуется не как тяжёлая enterprise-платформа, а как специализированная программа заполнения и формирования отчётных форм для большого числа предприятий. В этом варианте сохраняются React, PostgreSQL/Postgres Pro, Tauri для офлайн-клиента и доменная модель ОКО; меняется только серверный каркас.

Рекомендация: зафиксировать REST-контракты и доменные сервисы так, чтобы выбор NestJS или Fastify не влиял на фронтенд, БД и формат обмена. Для ТЗ основным вариантом считается NestJS, Fastify — альтернативный путь с меньшими трудозатратами на миграцию backend.
Допустим переходный этап: API на Express с последующей миграцией на NestJS или Fastify без смены контрактов REST.

9. Интеграция и форматы обмена

Версионируемый формат обмена комплектом между десктопом, порталом и ЦО. Поля: version, organization, periodStart, periodEnd, zid, eid, instances[], опционально rules{} для офлайн.

/api/auth — вход, SSO callback
/api/organizations, /api/periods

/api/kontr, /api/saldo, /api/audit
9.3. Миграция из MDB
Скрипты Python (generate_schemas_from_mdb.py, export_mdb_data.py) или сервис импорта на API для загрузки z261.mdb в PostgreSQL при обновлении методологии каждый квартал.

13. Критерии приёмки
Заполнение всех 76 форм комплекта за отчётный период.
Выполнение активных увязок period_active с результатом, сопоставимым с Access на эталонном MDB (допуск по согласованной матрице).
Перенос сальдо: совпадение с эталоном на тестовой паре периодов.
Импорт ReportPackage из десктопа в ЦО без потери данных.
Экспорт Excel по форме N01_1 (и согласованному перечню) — визуальное соответствие шаблону Минфина.
Роли: исполнитель не редактирует увязки; admin — редактирует.
Офлайн: 10 пользователей на одном комплекте без потери данных (нагрузочный сценарий).

А.5. Влияние выбора Fastify на оценку
Если вместо NestJS выбрать Fastify, целевой функционал не меняется. Экономия возникает только на миграции текущего Express API и на количестве архитектурных абстракций backend. Ориентировочно можно уменьшить этап 1 на 120–160 ч и общий резерв на 20–30 ч. Итоговая оценка для варианта Fastify: около 4760–4810 ч с резервом. При этом для большой команды NestJS остаётся более строгим и управляемым вариантом.

Приложение Б. Макеты экранов (логическая структура UI)
Ниже — текстовые макеты ключевых экранов для согласования с Заказчиком. Визуальный дизайн (цвета, шрифты) — в корпоративном стиле Заказчика.
Б.1. Вход / SSO
Центр: логотип ОКО, поля логин/пароль или кнопка «Войти через корпоративный портал». Футер: версия системы. После входа — редирект на дашборд или «Мои формы».
Б.2. Дашборд комплектов (admin / coordinator)
Фильтры: организация, период. Таблица: ZID, EID, название, % заполнения, draft/submitted. Кнопки: «Создать комплект», «Импорт JSON», «Экспорт». Клик по строке → комплект.
Б.3. Каталог шаблонов
Слева: дерево разделов N01–N19, ND. Справа: карточки форм (код, название, «Создать экземпляр»). Поиск по коду/названию.
Б.4. Мои формы
Верх: фильтр периода, организации (admin). Группировка по периоду или пакету. Таблица: форма, статус, дата изменения, ZID/EID. Действия: открыть, удалить, импорт JSON.
Б.5. Редактор формы
Шапка: название формы, организация, период, ед. изм., статус draft/submitted. Центр: таблица ячеек (закреплённые заголовки, итоговые строки). Справа/снизу: панель проверок (ошибки/предупреждения), кнопки «Пересчёт», «Сальдо», «Excel», «PDF». Для расшифровок: кнопка [...] → справочник контрагентов.
Б.6. Сводка и импорт (Tools)
Блоки: полнота комплекта (76 форм), запуск проверок по комплекту, экспорт JSON/Excel пакета, импорт ReportPackage, агрегация.
Б.7. Редактор увязок (/admin/checks)
Слева: таблица правил (№, выражение, active). Фильтры: форма, поиск. Справа: редактирование выражения, сообщения, флагов; кнопки «Сохранить», «Проверить на данных».
Б.8. Конструктор форм (/admin/forms)
Выбор формы из каталога. Вкладки: «Графы», «Строки», «Превью». Табличное редактирование row_no, label, FTotal, column keys. Импорт из MDB.
Б.9. Редактор сальдо (/admin/saldo)
Таблица правил переноса: source form, target form, колонки, условия. Кнопка «Запустить перенос» с выбором файла-источника.
Б.10. Пользователи и аудит
Список пользователей, роли, привязка к ZID. Журнал: дата, пользователь, действие, сущность.
Б.11. Desktop: приветствие
Кнопки: «Открыть папку комплекта», «Создать комплект», «Импорт JSON от ЦО». Поле пути к последней папке.
Б.12. Desktop: список форм комплекта
Sidebar: метаданные комплекта (орг, период, ZID/EID). Список 76 форм со статусами. Координатор: «Назначения», «Экспорт в ЦО», «Backup».
Б.13. Desktop: редактор формы (совместный)
Как веб-редактор + индикатор синхронизации, занятые ячейки (цвет/имя пользователя), строка «кто в форме сейчас».
